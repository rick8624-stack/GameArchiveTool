# GameArchiveTool 开发文档

> 本文档整合《GameArchiveTool需求文档.md》与代码实现，记录架构设计、关键决策与模块细节，供后续迭代/维护参考。使用说明见 `README.md`；本文档只讲"为什么这么设计"和"怎么实现的"。

---

## 1. 项目定位

Windows 单机 GUI 工具，处理大量游戏压缩包（Xbox360 游戏等）的批量流水线：

```
文件名预处理（清理末尾多余字符）
        ↓
批量解压（分卷识别 + 密码池 + 断点续传）
        ↓
批量重命名（CSV 编号对照表：中文名 → 英文名）
```

技术栈：Python 3.10+、tkinter（标准库，无必需第三方依赖）、7z.exe 子进程调用。目标产物是 pyinstaller 打包的单文件 exe。

---

## 2. 整体架构

```
main.py                 入口：创建窗口、启动事件循环
gui.py                  App 类：UI 构建 + 线程调度 + 事件处理（唯一持有 tkinter 状态的地方）
utils.py                无状态工具函数：路径、非法字符清理、pyinstaller 资源路径
core/
  config.py             config.json 读写 + 密码池 CRUD
  progress_store.py      progress.json 断点续传
  sevenzip.py            7z.exe 子进程封装（唯一与外部进程交互的模块）
  preprocess.py          模块一：纯函数式的扫描 + 执行
  extract.py             模块二：纯函数式的扫描 + 执行（依赖 sevenzip / config）
  rename.py              模块三：纯函数式的扫描 + 执行
  report.py              处理报告 CSV 导出
```

**分层原则**：`core/` 下每个模块都不 import tkinter，也不直接打印/弹窗——所有对外反馈都通过 `log(msg, tag)` 回调函数和返回值（dataclass）传递。这样：

- 三个业务模块可以脱离 GUI 单独测试（构造好参数直接调用 `build_plans` / `execute_plans`）
- GUI 的职责被限制在"how to present"，不掺入"what to do"的业务逻辑
- 未来如果要加 CLI 模式或批处理脚本，`core/` 可以直接复用

---

## 3. 线程模型（GUI 不阻塞的核心机制）

tkinter 的规则：**只有主线程能碰 UI 控件**。批量解压/重命名动辄处理几百个文件、耗时数分钟，必须放到子线程，但子线程不能直接调用 `self.xxx_label.configure(...)`。

`gui.py` 的解法：

```
工作线程（core/* 的执行函数）
   │  self._wlog(msg, tag) / self._emit(type=..., ...)
   ▼
event_queue: queue.Queue()   ← 线程安全，无需额外加锁
   ▲
   │  root.after(100, self._poll_queue)  每 100ms 轮询一次
主线程（_poll_queue → _handle_event → 更新 tkinter 控件）
```

关键点：

- `_start_worker()` 保证同一时间只有一个批量任务在跑（`self.worker.is_alive()` 检查），避免用户重复点击导致状态错乱
- `_worker_wrapper()` 用 `try/except Exception` 兜底：任何未预料的异常都会被捕获记录，并保证 `finally` 里的 `done` 事件一定发出——否则一旦子线程死掉，UI 会永远卡在"运行中"状态（按钮全灰、无法再次操作）
- `stop_event: threading.Event()` 是唯一的跨线程停止信号，工作线程在每个文件处理完之后检查一次，因此"停止"是**当前文件处理完后停止**，不是强杀线程（强杀 subprocess 中的 7z 容易留下损坏的半解压文件）

## 4. 7z 子进程封装（`core/sevenzip.py`）

这是整个工具里最"坑多"的部分，几个非显而易见的决策：

| 决策 | 原因 |
|---|---|
| `stdin=subprocess.DEVNULL` | 7z 遇到需要密码但没给 `-p` 参数时会**交互式等待输入**，子进程会挂死。必须显式关闭 stdin，逼 7z 走非交互路径 |
| 即使密码为空也要传 `-p` | 同上，`-p""` 明确告诉 7z "不要问我要密码"，而不是省略参数 |
| `-sccUTF-8` | 7z 默认按系统代码页输出，中文文件名在解析进度行/错误信息时会乱码，强制 UTF-8 |
| 按**字符**读取输出流，而非按行 | `-bsp1` 的实时进度是用 `\b`（退格）在同一行原地刷新的，标准的按行读取（`readline()`）会永远等不到 `\n` 而卡住。代码里把 `\b`/`\r` 当作 `\n` 处理来切分"行" |
| 只保留头尾 300 行日志 | 超大压缩包的进度输出可能几万行，全部存内存会有压力 |
| `test()` 与 `extract()` 分离 | 见下节密码池验证流程 |

**失败原因分类**（`classify_failure`）优先级：缺分卷 > 密码错误 > 无法识别 > 分卷数据错误 > 未知损坏。其中"分卷数据错误"和"密码错误"在 7z 的输出上有时难以区分（加密分卷密码错时也可能报 CRC/data error），所以额外读取磁盘上同名 `.NNN` 分卷的实际数量，判断编号是否连续来辅助分类——这是需求文档里"疑似分卷不完整"判断逻辑的具体实现。

## 5. 密码池验证流程（`core/extract.py::extract_one`）

```
候选密码 = [""] + 密码池（按命中次数降序）
for pwd in 候选密码:
    result = 7z t archive -p{pwd}      # 只测试，不解压
    if result.ok:
        命中 = pwd; break
    reason = classify_failure(result)
    if reason 属于 "分卷不完整" 或 "无法识别":
        break   # 这类错误换密码也没用，提前放弃，节省时间
    # 否则（密码错误/CRC等）继续试下一个密码

if 未命中:
    记为失败
else:
    真正执行 7z x archive -p{命中}      # 验证通过才解压
    命中密码计数 +1 并持久化
```

**为什么先 `t` 再 `x`，而不是直接试解压**：如果直接用错误密码解压，7z 可能已经往目标目录写入了部分（损坏的）文件，污染目标目录；用 `t`（测试模式）验证密码不会产生任何文件系统副作用，确认无误后再真正解压，保证目标目录要么完整要么完全没动过。

密码池的命中计数是持久化学习机制：常用密码会因为 `sorted_passwords()`（按 hits 降序）逐渐排到前面，减少后续任务的平均尝试次数。

## 6. 分卷识别设计（`core/extract.py::find_archives`）

五种正则 + 匹配顺序是精心设计的，顺序错了会导致误判：

1. `_NNN_VOLUME_RE`：`xxx.7z.001` / `xxx.zip.001` 式数字分卷
2. `_BARE_PART_RE`：**必须排在 `_SINGLE_RE` 之前**——裸命名分卷（文件名就是 `part1.rar`）如果先被 `_SINGLE_RE` 当成"名为 part1 的独立压缩包"匹配掉，`part2.rar`/`part3.rar` 会被当成另外的独立压缩包重复处理
3. `_PART_VOLUME_RE`：`游戏名.part1.rar` 带主名的分卷
4. `_R_VOLUME_RE`：旧式分卷的后续卷 `.r00`/`.r01`，直接跳过（不作为独立条目，会在处理主 `.rar` 时被 `_collect_r_volumes` 一并收集）
5. `_SINGLE_RE`：普通单文件压缩包；命中后如果是 `.rar` 还要反查是否存在同名 `.r00`（旧式分卷主卷）

所有分卷类型只处理"首卷"（`.001`/`part1`/`part01`），后续卷交给 7z 自动串联读取——这是 7z 分卷解压的标准用法，代码里只是负责"别把后续卷也当独立任务重复跑一遍"。

`ArchiveItem.volume_files` 记录分卷组的**全部**文件（不只是首卷），是为了"解压成功后删除原压缩包"这个可选功能——必须把分卷全部删掉，而不是只删首卷。

## 7. 断点续传（`core/progress_store.py`）

极简设计：`progress.json` 只存一个已完成路径的集合，每完成一个压缩包立即 `mark_done()` 落盘一次（而不是等全部任务结束才写），这样即使进程被强杀/断电，已完成的部分也不会丢失。任务全部正常结束后 `clear()` 删除该文件——下次启动 `find_archives` 扫到的都是"新"任务，不会因为陈旧的 progress.json 而误跳过。

`gui.py` 里的交互：检测到 `progress.json` 非空时，用对话框询问用户"跳过已完成 / 清空重来"，而不是默默自动跳过——避免用户在换了根目录或对照表之后，被旧进度记录误导。

## 8. 配置持久化（`core/config.py`）与 pyinstaller 兼容（`utils.py`）

`Config.load()` 只覆盖 `DEFAULTS` 中已知的键，这样版本迭代新增配置项时，旧的 `config.json` 不会因为"缺键"而导致程序崩溃或该功能永远拿到 `None`；配置文件损坏（JSON 解析失败）时静默回退默认值，而不是让程序无法启动。

`utils.app_dir()` 是打包后能正常读写配置文件的关键：pyinstaller 单文件模式下运行时会把整个程序解压到一个临时目录（`sys._MEIPASS`），这个目录**只读且退出后自动清理**，`config.json`/`progress.json` 绝对不能往那里写。所以：

- 判断 `sys.frozen` 是否为真（pyinstaller 运行时会设置这个属性）
- 是则用 `Path(sys.executable).parent`（exe 所在目录，可写）
- 否则用 `Path(__file__).parent`（开发环境下的项目目录）

`utils.resource_path()` 是反过来的场景：读**只读资源**（如果未来加图标等静态资源）要去 `sys._MEIPASS` 找，两个函数不能混用。

## 9. 批量重命名的编号匹配（`core/rename.py`）

编号归一化 `normalize_code()` 把 `"001"` → `"1"`，`"XB001"` → `"XB1"`：字母前缀转大写、数字去前导零，这样文件夹里的 `001.xxx` 能匹配 CSV 里写的 `1`（用户手工填表时通常不会特意补前导零）。

`build_plans()` 按路径深度**降序**排序文件夹（`len(p.parts)` 从大到小），保证自底向上重命名——如果先改了父文件夹的名字，子文件夹的原路径就失效了，后续基于旧路径的操作会失败。

CSV 读取用 `utf-8-sig`（而不是 `utf-8`）是为了兼容 Excel 导出 CSV 时默认带的 BOM 头，否则第一行第一个字段会混入不可见字符导致匹配失败。

## 10. 已知边界情况处理一览

| 情况 | 处理方式 |
|---|---|
| 压缩包内文件与目标目录已有文件重名 | `7z x -y` 直接覆盖，同时在日志里 warn 标注，不静默覆盖 |
| 分卷缺失（如只有 .001 没 .002） | 不归为单纯"密码错误"或"损坏"，标注"疑似分卷不完整"，附带已找到的分卷数 |
| 路径超过 Windows 260 字符限制 | `is_path_too_long()` 检查后跳过并警告，不尝试处理 |
| 压缩包本身损坏 | 归入失败清单，classify_failure 尽量给出可读原因 |
| 空文件夹 / 无压缩包的文件夹 | `rglob("*")` 天然不产生条目，静默跳过，不刷无意义日志 |
| 重命名目标已存在 | 预处理和重命名模块都会在生成计划阶段标记 `skip=True` 并注明原因，而不是执行时才失败 |

## 11. 报告导出（`core/report.py`）

`ReportRecord` 是三个模块共用的统一记录格式（路径/操作类型/结果/详情/密码/耗时），`gui.py` 在每个模块执行完后都往同一个 `self.report_records` 里追加，因此导出的 CSV 是**跨模块的完整会话记录**，不是分模块各自导出。编码用 `utf-8-sig`，保证双击用 Excel 打开中文不乱码（这一点和读 CSV 对照表时用 `utf-8-sig` 是对称的）。

## 12. 已验证的架构扩展点（对应 README「后续迭代方向」）

- **游戏库管理**（扫描解压后文件夹生成清单）：可在 `core/` 下新增模块，复用现有的 `log(msg, tag)` 回调约定和 `event_queue` 事件类型（加一个新的 `type=` 分支），`gui.py` 加一个新 tab 即可，不需要改动现有三个模块
- **NSZ/XCI 等格式转换**：`core/sevenzip.py` 的"子进程封装 + 按字符解析进度 + stdin=DEVNULL"这套模式可以整体复制给其他命令行转换工具，只需替换 `_run()` 的 `cmd` 构造
- **多任务并行解压**：当前 `gui.py` 的 worker 是单线程顺序处理（`self.worker: threading.Thread | None`，同一时间只允许一个）。要改成线程池，`event_queue` 天然支持多生产者单消费者，不需要改 UI 侧的 `_poll_queue`；但 `progress_store` 和 `config.record_hit()` 目前不是线程安全的（并发写 JSON 会互相覆盖），并行化之前必须先给这两处加锁或改成单写者模式
