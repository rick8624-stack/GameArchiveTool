# GameArchiveTool - 游戏压缩包批量处理工具

Windows GUI 工具，对多层文件夹中的游戏压缩包（Xbox360 游戏等）完成
**文件名预处理 → 批量解压（密码池）→ 文件夹批量重命名** 的完整流水线。

## 运行环境

- Windows，Python 3.10+
- 已安装 [7-Zip](https://www.7-zip.org/)（默认路径 `C:\Program Files\7-Zip\7z.exe`，可在界面中修改）
- GUI 基于标准库 tkinter，**无必需第三方依赖**；可选安装 `tkinterdnd2` 启用拖拽文件夹进窗口：

```bash
pip install -r requirements.txt   # 可选，仅为启用拖拽
python main.py
```

## 使用说明

主窗口只保留最常用的内容：**源目录**（支持拖入文件夹）、**目标目录**（留空=原地解压）、
**解压密码池**，以及三个一键按钮。引擎路径（7z/UnRAR）、清理规则、各类开关、CSV 对照表
等细节收纳在 **详细设置** 对话框中（设置自动保存）。所有耗时操作都在后台线程执行，
可随时点击进度区的 **停止** 按钮中断（当前文件处理完后停止）。

- **一键解压**：文件名清理 → 伪装扩展名修正 → 批量解压（密码池 + WinRAR 回退 +
  嵌套解压 + 断点续传），即下文的完整流水线
- **一键预处理**：智能统一同一文件夹内命名不一致的分卷。典型场景：下载时各卷被加了
  不同后缀（`游戏A.7z.001` / `游戏A(1).7z.002` / `游戏A - 副本.7z.003`）导致 7z 无法
  串联，按首卷主名统一。判定条件保守：仅当同目录同类型分卷卷号从 1 连续且无重复时
  才动手（先在日志中预览，确认后执行）
- **一键重命名**：一级子文件夹按名称排序改为 `1, 2, 3...`（默认编号模式），
  可用「回退重命名」恢复；对照表模式（编号→英文名）在详细设置中
- **一键压缩**：把源目录下**每个一级子文件夹**分别压缩成单独的压缩包
  （`游戏A/` → `游戏A.7z`），包内顶层即文件夹名，正好与解压侧「唯一顶层
  文件夹」约定形成闭环。格式（7z/zip）、压缩级别、密码、分卷大小、输出目录、
  压缩后删源等在详细设置中。跳过模式下目标包已存在则跳过；空文件夹自动跳过

### 1. 批量解压（含解压前文件名清理）

**文件名清理**（原预处理模块，已并入解压流水线）：去掉文件名末尾的多余字符
（默认"删"字，如 `游戏.7z.001删` → `游戏.7z.001`）。

- 支持多条规则（`;` 分隔，如 `删;.bak`），勾选"按正则解释"后按正则匹配（自动锚定末尾）
- 勾选"解压前自动清理"（默认开）后，点"开始批量解压"会先清理再扫描解压，一步到位
- 也可单独使用：**预览清理**（在日志中列出 原名 → 新名）→ **执行清理**

**解压**：递归扫描所有压缩文件并逐个解压：

- **智能识别伪装扩展名**（默认开）：按文件头魔数识别内容是 7z/rar/zip 但扩展名
  不对的文件（如 `截图.jpg` 实为 zip），自动修正为 `截图.jpg.zip` 后正常解压。
  多重防误判：**分卷感知**——分卷成员及 SFX 首卷（`game.exe` 配 `game.001`）
  一律不改名，绝不把分卷首卷变成独立压缩包而拆散分卷；docx/apk/jar 等 zip 容器
  格式排除；`.save`/`.sav`/`.pak` 等游戏存档与引擎资源格式排除；**小于阈值
  （默认 1 MB，界面可调）的文件不识别**——游戏存档通常只有 1KB–100KB 且以 zip
  存储，靠大小门槛避免被误解压。此外解压前会**自动修复**曾被错误追加扩展名而
  拆散的分卷（`game.001.7z` → `game.001`，当同组其他卷存在时）
- **目标解压路径**：可指定输出目录，压缩包按相对根目录的文件夹结构解压过去；
  留空则解压到压缩包所在目录（原行为）
- **嵌套解压**（默认开）：解压产物里发现新压缩包（含伪装扩展名的）自动继续解压，
  最多 4 层，防止无限套娃

- 支持 zip / rar / 7z / tar（含 `.tar.gz`/`.tgz`/`.tar.bz2`/`.tar.xz` 及 gz/bz2/xz
  单文件压缩，均由 7z 直接解压；`.tar.gz` 先解出 `.tar` 再经嵌套解压自动解包），
  以及分卷：`.7z.001`、`.zip.001`、`游戏名.part1.rar`、
  裸命名的 `part1.rar`/`part01.rar`（按所在文件夹归组）、`.rar`+`.r00` 旧式 rar 分卷、
  `.zip`+`.z01` 旧式 zip 分卷、`游戏.001` 无格式中缀纯数字分卷。
  分卷只处理首卷，后续卷由 7z 自动串联
- **WinRAR 引擎回退**：部分 WinRAR 新版本生成的加密 rar，7z 报错但 WinRAR 能解。
  7z 全部密码尝试失败且目标是 rar 系文件时，自动改用 UnRAR.exe（路径可配，
  默认自动探测 WinRAR 安装目录，未安装则不回退）重试密码池并完成解压
- **强制解压**（默认关）：**未加密**压缩包因数据损坏导致校验失败时，用已确认
  无密码问题的密码强制解压，保留已解出的部分内容并记为成功（结果可能不完整），
  且**不再回退 UnRAR**。适用于确定原包损坏但仍想抢救出可用文件的场景。
  安全约束：**加密包 + 密码不匹配时绝不强制解压**——用错误密码只会产出一堆
  0KB 垃圾文件；这种情况直接记为「密码错误」失败，且不会删除原压缩包
- **跳过模式**（默认开）：目标位置已存在同名文件夹（子文件夹模式的目标夹，
  或压缩包内唯一顶层文件夹）且非空时，跳过该压缩包——重复运行不会重复解压
- **密码池**：先尝试无密码，再按命中次数从高到低尝试；先用 `7z l` 快速验证
  （头部加密的包错误密码立即排除，免整包测试），通过后再 `7z t` 完整校验，
  校验通过才真正解压；命中的密码计数 +1 并持久化，常用密码自动前移；
  批次内上一个包命中的密码会优先尝试（同一来源的游戏包通常共用密码）
- 默认解压到压缩包所在文件夹，可勾选"解压到以压缩包名命名的子文件夹"；
  包内已有唯一顶层文件夹时自动跳过子文件夹，避免 `游戏A/游戏A/` 双重嵌套
- **磁盘空间预检**：按包内未压缩总大小对比剩余空间，不足则记失败不动手
- 可勾选"解压成功后删除原压缩包"（含分卷全部文件），默认关闭；
  安装了 `send2trash` 时删除进回收站（可恢复），否则永久删除
- 失败的压缩包记入失败清单（区分密码错误 / 压缩包损坏 / 疑似分卷不完整），
  不中断整体流程；结束后可点 **重试失败项** 只重跑失败的（如补了密码之后）
- **断点续传**：每完成一个压缩包即写入 `progress.json`；中途退出后再次开始解压时，
  会询问是否跳过已完成项
- 任务结束后可点 **导出处理报告 CSV**（路径、操作类型、结果、使用的密码、耗时）

### 2. 批量重命名

两种模式（界面中单选），所有执行过的重命名都会记入根目录下的
`.rename_history.json`，随时可点 **回退上次重命名** 恢复（可逐批往回退）。

**默认编号模式**：无需对照表，把根目录下的一级子文件夹按名称排序，
依次重命名为 `1, 2, 3...`。

**对照表模式**：读取 CSV 对照表（两列：`编号,英文名`，UTF-8 编码，兼容 Excel
导出的带 BOM 文件），把 `001.中文名` 式文件夹改名为 `001.英文名`：

- 编号部分保留原样（含前导零），匹配时自动去前导零（文件夹 `001` 匹配对照表的 `1`）
- 支持字母编号（`XB001`、`K001`、`KX01` 等，字母大小写不敏感）
- 新文件夹名自动清除 Windows 非法字符 `\/:*?"<>|`
- 未匹配到对照表的文件夹显示在右侧"待处理清单"
- 同样是先 **预览** 后 **执行**

## 配置文件

程序目录下自动生成（打包后位于 exe 同目录）：

- `config.json`：7z 路径、密码池（含命中计数）、上次使用的根目录、各开关状态
- `progress.json`：批量解压的断点续传记录，任务全部完成后自动清空
- `<根目录>/.rename_history.json`：文件夹重命名历史（跟数据走），供"回退上次重命名"使用

## pyinstaller 打包

```bash
pip install pyinstaller

# 未安装 tkinterdnd2（无拖拽）：
pyinstaller --onefile --windowed --name GameArchiveTool main.py

# 已安装 tkinterdnd2（带拖拽，需要收集其 tkdnd 二进制文件）：
pyinstaller --onefile --windowed --name GameArchiveTool --collect-all tkinterdnd2 main.py
```

生成的单文件 exe 位于 `dist\GameArchiveTool.exe`。
注意：exe 需要放在**可写目录**运行（config.json / progress.json 保存在 exe 旁边）。

## 项目结构

```
main.py                 入口
gui.py                  主窗口、线程/事件队列、进度与日志
utils.py                路径工具、非法字符清理、_MEIPASS 兼容
core/
  config.py             config.json 与密码池管理
  progress_store.py     progress.json 断点续传
  sevenzip.py           7z.exe 子进程封装、-bsp1 进度解析、失败分类
  preprocess.py         模块一：文件名清理
  extract.py            模块二：批量解压
  rename.py             模块三：编号对照表重命名
  compress.py           模块四：一键压缩（每个一级子文件夹→单独包）
  report.py             处理报告 CSV 导出
```

## 测试与 CI/CD

本地运行测试（无第三方依赖，找不到 7z.exe 时解压流程测试自动跳过）：

```bash
python -m unittest discover -s tests -v
```

GitHub Actions 工作流：

- **CI**（`.github/workflows/ci.yml`）：push 到 main 或提 PR 时，在 windows-latest
  上用 Python 3.10 / 3.12 双版本跑语法检查与全部测试（运行器预装 7-Zip，
  解压流程为真实执行）
- **CD**（`.github/workflows/release.yml`）：推送 `v*` 标签时自动测试 → pyinstaller
  打包单文件 exe → 发布 GitHub Release 并附上 `GameArchiveTool.exe`：

  ```bash
  git tag v1.1.0
  git push origin v1.1.0
  ```

  也可在 GitHub 的 Actions 页面手动触发打包验证（只出构建产物，不发 Release）。
